# OWASP Top 10:2025 — Web Rulebook

For each category: **Discover** patterns to grep, **Verify** questions to confirm a real flaw, **Severity baseline** to start from (adjust per impact), **Fix** template.

Reference: https://owasp.org/Top10/2025/

---

## A01:2025 — Broken Access Control

Includes IDOR, missing authz on endpoints, path traversal, CORS-driven access, and SSRF (folded in for 2025).

### Discover
- Route definitions: `(app|router)\.(get|post|put|delete|patch)`, `@app.route`, `@router\.`, `Route::`, `gin\.(GET|POST)`, `http\.HandleFunc`, `def \w+\(request`
- File path operations using user input: `path\.join.*req\.`, `open\(.*request\.`, `fs\.read.*params`, `..` traversal patterns
- Outbound HTTP using user input: `requests\.get\(.*req`, `axios\.get\(.*req\.`, `fetch\(.*req\.`, `http\.Get\(.*r\.`
- CORS config: `Access-Control-Allow-Origin`, `cors\(`, `CORS_`

### Verify (per candidate)
1. Does an auth middleware run before this handler? (`requireAuth`, `@login_required`, `authenticate`, framework guard)
2. For object-scoped routes (`/users/:id`, `/orders/:id`): does the handler check the resource belongs to the caller?
3. For admin routes (`/admin/*`, `/internal/*`): is there a role/permission check beyond authentication?
4. For file paths: is the user input normalized and validated against an allowlist or base directory?
5. For outbound requests: is the URL validated against a domain allowlist? Are private IP ranges (10/8, 172.16/12, 192.168/16, 169.254/16, 127/8, ::1, fc00::/7) blocked? Are redirects followed without re-validation?
6. CORS: is `Access-Control-Allow-Origin: *` paired with `Allow-Credentials: true`? Is the allowed origin reflected from the request without validation?

### Severity baseline
- Missing auth on data-modifying endpoint → **Critical**
- IDOR on sensitive data (PII, billing, messages) → **Critical**
- IDOR on low-sensitivity data → **High**
- Missing role check, auth present → **High**
- Path traversal with file write → **Critical**
- Path traversal read-only → **High**
- SSRF to internal network → **High** (Critical if cloud metadata reachable)
- CORS wildcard with credentials → **High**

### Fix template
- Add `requireAuth` + `requireRole(...)` middleware.
- For IDOR: `if (resource.ownerId !== req.user.id) return 403`.
- For path traversal: resolve absolute path, assert it starts with the allowed base dir.
- For SSRF: validate URL scheme (https only), resolve hostname, reject private IPs, disable redirects or re-validate after each hop.
- For CORS: pin to specific origins, never reflect, never combine `*` with credentials.

## A02:2025 — Security Misconfiguration

Moved up from #5 to #2 — reflects breach reality.

### Discover
- Config files: `.env*`, `config/*.{yml,yaml,json,toml}`, `settings.py`, `application.{properties,yml}`, `wp-config.php`, `appsettings.json`
- Debug flags: `DEBUG\s*=\s*True`, `app\.debug\s*=\s*true`, `NODE_ENV.*development`, `display_errors.*On`
- Default creds: `admin:admin`, `root:root`, `password.*=.*"password"`
- Verbose error handlers: `traceback`, stack-trace-returning error responses
- Cloud configs: `s3:*`, IAM `Action: "*"`, security group `0.0.0.0/0`
- Cookie flags: `httpOnly`, `secure`, `sameSite` settings on session cookies
- Security headers: presence of `Content-Security-Policy`, `X-Frame-Options`, `Strict-Transport-Security`, `X-Content-Type-Options`

### Verify
1. Is `DEBUG=true` / `NODE_ENV=development` actually shipped to production? (check Dockerfile, deploy scripts, env templates)
2. Are error responses returning stack traces or DB error messages to clients?
3. Are session cookies missing `httpOnly`, `secure`, or `sameSite`?
4. Are security headers absent from responses?
5. Are cloud IAM policies using `"*"` for actions or resources unnecessarily?
6. Are admin panels (`/admin`, `/phpmyadmin`, `/.env`, `/.git`) reachable without auth?
7. Are dev dependencies / debug tools in the production bundle?

### Severity baseline
- `DEBUG=true` in production with stack traces → **High**
- Default admin creds → **Critical**
- Missing `httpOnly` on session cookie → **Medium**
- Missing CSP on user-content pages → **Medium**
- IAM wildcard on production resources → **High**
- `.env` / `.git` exposed via web → **Critical**

### Fix template
- Force `DEBUG=false` via env-driven config; gate stack traces behind env check.
- Add session cookie flags: `httpOnly: true, secure: true, sameSite: 'lax'` minimum.
- Add security header middleware (helmet, secure_headers, etc.).
- Tighten IAM to least-privilege specific actions/resources.

## A03:2025 — Software Supply Chain Failures

NEW for 2025. Broader than "Vulnerable Components" — covers the whole build/distribute/update chain.

### Discover
- Dependency manifests: `package.json`, `package-lock.json`, `yarn.lock`, `pnpm-lock.yaml`, `requirements.txt`, `Pipfile.lock`, `poetry.lock`, `go.mod`, `go.sum`, `Cargo.toml`, `Cargo.lock`, `Gemfile.lock`, `composer.lock`, `pom.xml`, `build.gradle`
- CI configs: `.github/workflows/*.{yml,yaml}`, `.gitlab-ci.yml`, `.circleci/config.yml`, `Jenkinsfile`
- Container files: `Dockerfile`, `docker-compose*.{yml,yaml}`
- Install scripts in deps: `preinstall`, `postinstall`, `prepare` in package.json deps
- Untrusted registries: `registry =`, `index-url`, custom repository URLs
- SBOM presence: `sbom.{json,xml}`, `bom.{json,xml}`, `cyclonedx*`

### Verify
1. Is there a lockfile committed? Is it out of sync with the manifest?
2. Are dependencies pinned to specific versions, or using ranges (`^`, `~`, `*`, `latest`)?
3. Run `npm audit` / `pip-audit` / `bundle audit` / `cargo audit` / `osv-scanner` if available — count critical/high.
4. Do any dependencies have `postinstall` / `preinstall` scripts? Are they from trusted sources?
5. Are CI tokens / secrets scoped to minimum permissions? Any `permissions: write-all`?
6. Does CI use third-party actions pinned to commit SHAs, or to mutable tags (`@v1`, `@main`)?
7. Are container base images pinned by digest (`@sha256:...`) or by tag (`:latest`, `:v1`)?
8. Is there an SBOM? Are artifacts signed (cosign, sigstore)?
9. Are dependencies fetched from internal mirrors / proxies, or directly from public registries?

### Severity baseline
- Critical CVE in production dependency → **Critical**
- High CVE in production dependency → **High**
- No lockfile → **Medium**
- CI action pinned to mutable tag → **Medium**
- `permissions: write-all` in CI → **High**
- Untrusted `postinstall` script → **High**

### Fix template
- Commit lockfile; pin versions; renovate/dependabot for managed updates.
- Pin CI actions to commit SHAs.
- Pin container base images by digest.
- Scope `GITHUB_TOKEN` permissions per workflow.
- Add SBOM generation and artifact signing.

## A04:2025 — Cryptographic Failures

### Discover
- Hashing: `md5`, `sha1`, `MessageDigest.*MD5`, `hashlib\.md5`, `crypto\.createHash\(['"]md5`
- Password hashing: `bcrypt`, `argon2`, `scrypt`, `PBKDF2`, plain `sha256\(.*password`
- Encryption: `AES.*ECB`, `DES`, `RC4`, `\.createCipher\(` (deprecated), hardcoded IVs
- Random: `Math\.random`, `random\.random` used for tokens/keys (vs `secrets`, `crypto.randomBytes`)
- TLS: `verify=False`, `rejectUnauthorized: false`, `ssl._create_unverified_context`, `InsecureSkipVerify: true`
- Hardcoded keys: hex strings ≥32 chars, `BEGIN .* PRIVATE KEY`, `-----BEGIN`

### Verify
1. Are passwords hashed with bcrypt/argon2/scrypt/PBKDF2 (not md5/sha1/sha256-bare)?
2. Are session tokens / API keys generated with a CSPRNG?
3. Is sensitive data encrypted at rest? With what algorithm and key management?
4. Is TLS verification disabled anywhere in production code paths?
5. Are private keys / API keys hardcoded or committed to the repo? (also covered by secrets scanning)
6. Is `Math.random` / `random.random` used to generate any security token, password reset link, or session ID?

### Severity baseline
- Plaintext password storage → **Critical**
- MD5/SHA1 password hashing → **Critical**
- Hardcoded secret in repo → **Critical** (rotate immediately)
- TLS verification disabled in prod → **High**
- `Math.random` for security token → **High**
- Weak symmetric cipher (DES, RC4, AES-ECB) → **High**

### Fix template
- Migrate password storage to argon2id (or bcrypt cost ≥12).
- Use `crypto.randomBytes(32).toString('hex')` / `secrets.token_urlsafe(32)`.
- Move secrets to env vars / secret manager; rotate any committed key.
- Re-enable TLS verification; pin certificates if needed.

## A05:2025 — Injection

SQLi, NoSQLi, XSS, command injection, LDAP, template injection, header injection.

### Discover
- SQL string concatenation: `SELECT.*\+.*req\.`, `f"SELECT.*\{`, `"WHERE.*"\s*\.\s*\$`, `query\(.*\+\s*\w+`
- Raw query methods: `\.query\(`, `\.execute\(`, `\.exec\(`, `\.raw\(`, `cursor\.execute`
- Shell exec with input: `exec\(.*req`, `system\(.*\$`, `subprocess.*shell=True`, `child_process.*req`, `os\.system\(.*request`
- Template injection: `\${.*req\.`, `<%=.*params`, Mustache/Handlebars dynamic templates from user input
- DOM XSS: `innerHTML\s*=`, `dangerouslySetInnerHTML`, `v-html`, `document\.write`, `eval\(`
- Server XSS: response rendering of unescaped input; templates with `{{{ }}}` (raw) instead of `{{ }}` (escaped)
- LDAP: `ldapsearch.*\+`, `ldap.*filter.*\$`
- NoSQL: `\$where:.*req`, `find\(.*req\.body`, `MongoClient.*req`

### Verify
1. For each SQL call: is it parameterized (placeholders, prepared statements, ORM with binding)?
2. For shell exec: is `shell=True` set? Are arguments passed as a list, not a string?
3. For template rendering: is the template engine auto-escaping enabled? Are any blocks marked safe/raw containing user input?
4. For DOM: does user-controlled data reach `innerHTML` / `dangerouslySetInnerHTML` without sanitization?
5. For NoSQL: are operator injections possible (e.g. user passes `{$ne: null}` as a value)?
6. For headers: is user input flowing into `setHeader` / `Location` redirects without CRLF stripping?

### Severity baseline
- SQLi (any) → **Critical**
- Command injection with shell=True → **Critical**
- Stored XSS → **High**
- Reflected XSS → **High**
- DOM XSS → **High**
- SSTI (server-side template injection) → **Critical**
- NoSQL operator injection → **High**
- Open redirect → **Medium**

### Fix template
- Use parameterized queries everywhere. Reject `f"... {var} ..."` SQL strings.
- For shell: `subprocess.run([...], shell=False)` with arg list.
- For templates: escape by default; never mark user input as safe.
- For DOM: sanitize with DOMPurify or use `textContent` instead of `innerHTML`.
- For NoSQL: validate input types; reject objects where strings are expected.

## A06:2025 — Insecure Design

Missing controls at the design level — rate limits, anti-automation, business logic flaws.

### Discover
- Rate limit middleware: `express-rate-limit`, `flask-limiter`, `django-ratelimit`, `rack-attack`, `golang.org/x/time/rate`
- Auth-sensitive endpoints: `/login`, `/register`, `/password/reset`, `/2fa`, `/verify`
- Lockout logic: failed-attempt counters, account locking
- Captcha / anti-bot: reCAPTCHA, hCaptcha, Turnstile
- Workflow state machines: order/payment/checkout flows

### Verify
1. Is there a rate limit on `/login`? Per-IP? Per-account? What are the thresholds?
2. Is there a rate limit on `/password/reset` to prevent enumeration / spamming?
3. Does failed login lock the account, or just slow it down? Is there an unlock mechanism?
4. Is there CAPTCHA after N failed attempts?
5. For business flows: can a user skip steps? Replay step N after completing step N+1?
6. Is there protection against parallel requests racing the same resource (e.g. two requests redeeming the same coupon)?
7. Are pagination / search / export endpoints rate-limited to prevent scraping?

### Severity baseline
- No rate limit on login → **High**
- No rate limit on password reset → **High**
- No anti-automation on signup (allowing bot registration at scale) → **Medium**
- Race condition in payment/coupon flow → **High**
- Workflow step skipping → **High** (Critical if bypasses payment)

### Fix template
- Add per-IP and per-account rate limit on `/login` (e.g. 5/min per account, 20/min per IP).
- Add account lockout with exponential backoff or lockout-with-email-unlock.
- Add CAPTCHA after 3 failed attempts.
- Use database transactions / row locks for payment-affecting flows.

## A07:2025 — Authentication Failures

### Discover
- Password validation: minimum length, complexity, common-password lists
- JWT usage: `jwt\.(sign|verify|decode)`, `jsonwebtoken`, `pyjwt`, `jose`
- Session config: session lifetime, regeneration on login, fixation protection
- MFA: 2FA / TOTP / WebAuthn presence
- Password reset: token generation, expiry, single-use enforcement

### Verify
1. JWT: is `algorithm: 'none'` accepted? Is the algorithm pinned (not user-controlled)? Is the secret a strong random value? Is `exp` claim verified?
2. Sessions: regenerated on login (fixation protection)? Reasonable expiry? Invalidated on logout?
3. Passwords: minimum length (≥12 modern), checked against common-password list (e.g. haveibeenpwned API)?
4. Password reset: tokens use CSPRNG, expire (≤1h), single-use, sent to verified email?
5. Is MFA available? Required for admin? Backup codes implemented securely?
6. Is there account enumeration via login error messages, password reset responses, or signup ("email already exists")?
7. Are credentials transmitted over HTTPS only?

### Severity baseline
- JWT `alg: none` accepted → **Critical**
- JWT secret weak / hardcoded → **Critical**
- Password reset token reusable → **High**
- Password reset token doesn't expire → **High**
- No session regeneration on login → **High**
- Account enumeration via login → **Medium**
- No MFA option for admin → **Medium**

### Fix template
- Pin JWT algorithm in verify call (`{algorithms: ['HS256']}` or `RS256`).
- Generate reset tokens with `crypto.randomBytes(32)`; store hash in DB; mark used after first redemption.
- Regenerate session ID on login.
- Return identical responses for "wrong password" and "no such user".

## A08:2025 — Software & Data Integrity Failures

### Discover
- Deserialization: `pickle\.loads`, `yaml\.load[^_]`, `unserialize`, `Marshal\.load`, `JSON\.parse.*req` flowing into `eval`
- Auto-update / dynamic loading: `require\(.*req`, `import.*\$\{`, plugin loaders
- CI/CD pipelines: artifact signing, build provenance
- Webhook validation: signature verification on incoming webhooks (Stripe, GitHub, Slack, etc.)

### Verify
1. Is any user-controlled data deserialized with an unsafe deserializer (pickle, yaml.load without SafeLoader, PHP unserialize)?
2. Are webhook endpoints verifying HMAC signatures using a constant-time compare?
3. Are auto-updates / dynamic plugin loads verifying signatures?
4. Are CI artifacts signed before deploy?

### Severity baseline
- Pickle/yaml.load on user input → **Critical** (RCE)
- Missing webhook signature verification → **High**
- Unsigned auto-update → **Critical**
- Non-constant-time HMAC compare → **Medium**

### Fix template
- Replace `pickle.loads` with `json.loads` or signed pickle.
- Use `yaml.safe_load` instead of `yaml.load`.
- Verify webhook signatures with `hmac.compare_digest` (constant-time).
- Sign artifacts with cosign; verify on deploy.

## A09:2025 — Security Logging & Alerting Failures

### Discover
- Logging libraries: `winston`, `pino`, `bunyan`, `logging`, `logrus`, `zap`, `slog`
- Auth event logging: searches near login, logout, password change, MFA setup
- Sensitive data in logs: `log.*password`, `log.*token`, `log.*ssn`, `log.*credit`
- Alert/monitoring: Sentry, Datadog, PagerDuty integrations

### Verify
1. Are auth failures logged? Successes? With timestamp, IP, user ID?
2. Are admin actions logged with audit trail?
3. Are passwords / tokens / PII accidentally logged?
4. Are logs centralized (not just local stdout)?
5. Are there alerts on suspicious patterns (mass login failures, privilege escalations)?

### Severity baseline
- Passwords / tokens in logs → **High**
- No auth event logging → **Medium**
- No audit trail for admin actions → **Medium**
- No alerting on security events → **Low**

### Fix template
- Add structured logging for auth events with redaction of sensitive fields.
- Centralize logs to SIEM / log aggregator.
- Set up alerts on rate-limit-trip, mass-401, privilege change events.

## A10:2025 — Mishandling of Exceptional Conditions

NEW for 2025. Often under-audited.

### Discover
- Bare excepts: `except:`, `except Exception: pass`, `catch \(.*\) \{\s*\}`, `catch\(_\)`, `rescue\s*\n`
- Unhandled promise rejections: `\.then\([^)]+\)` without `.catch`, missing `await` on async, `Promise\.\w+` without rejection handler
- Network calls without timeouts: `fetch\(`, `axios\.`, `requests\.`, `http\.Get`, `http\.Client\{` — check for timeout config
- Panic-on-bad-input: type assertions / unwraps on user input
- Verbose error responses: `JSON.stringify(err)`, returning `err.stack` to client
- Resource leaks: missing `finally`, missing `defer`, unclosed file handles

### Verify
1. Are external network calls (DB, HTTP, cache, queue) wrapped with explicit timeouts?
2. Are exceptions silently swallowed anywhere in the request path?
3. Do error responses to clients leak stack traces, file paths, query strings, or internal IDs?
4. Are there circuit breakers or retry-with-backoff on flaky upstream services?
5. Do destructive operations (DROP, DELETE, rm) have safeguards (transactions, confirmation, rate limit)?
6. Are panics / unwraps reachable from user input?

### Severity baseline
- Stack trace leaked in error response → **Medium** (High if it reveals secrets/paths)
- Bare except swallowing security-critical errors (auth check, signature verify) → **High**
- No timeout on upstream calls (DoS risk) → **Medium**
- Unwrap / panic reachable from user input → **Medium** (High if affects multi-tenant service)
- Failed-open on auth check exception → **Critical**

### Fix template
- Replace bare `except:` with specific exception types; log + re-raise security-critical paths.
- Add timeouts to all external calls (sane defaults: 5s connect, 30s read).
- Strip stack traces from production error responses; map to generic 500 with correlation ID.
- Wrap unwraps with explicit error returns.
- For auth checks: fail-closed on exception (deny access) — never fail-open.
