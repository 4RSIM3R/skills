# OWASP Top 10 for LLM Applications:2025 — LLM Rulebook

For each category: **Discover** patterns to grep, **Verify** questions to confirm a real flaw, **Severity baseline**, **Fix** template.

Reference: https://genai.owasp.org/resource/owasp-top-10-for-llm-applications-2025/

---

## LLM01:2025 — Prompt Injection

Direct (user instructions override system) and indirect (instructions hidden in tool/RAG content).

### Discover
- LLM call sites: `anthropic\.`, `openai\.`, `\.messages\.create`, `\.chat\.completions`, `langchain`, `llm\.invoke`
- System prompts being concatenated with user input
- RAG pipelines: vector search results passed into prompts
- Tool/agent setups: tools that fetch web pages, read files, query DBs
- Output processing: LLM response parsed as JSON, code, SQL, shell commands

### Verify
1. Does any system prompt template directly concatenate user input before sending? (vs separating into user role)
2. For RAG: is retrieved content presented to the model with clear delimiters indicating it's untrusted data, not instructions?
3. For tools that fetch web/email/PDFs: is the fetched content shown to the model with markup distinguishing it from system instructions?
4. Is there an output filter / structured output mode that rejects "ignore previous instructions" patterns? (Note: filtering alone is weak — separation is the real defense.)
5. For multi-step agents: can a tool result inject new instructions that the model then follows?
6. Are user-controllable strings interpolated into the system prompt (vs the user message)?

### Severity baseline
- User input directly placed in system prompt → **High**
- Indirect injection via untrusted RAG/tool content with no separation → **High**
- LLM output flowing to destructive tool without confirmation → **Critical** (also LLM06)
- Agent following injected instructions to exfiltrate data → **Critical**

### Fix template
- Keep system prompt static; pass user content only via the `user` role.
- Wrap untrusted content (RAG hits, tool results) in clear markup like `<untrusted_content>...</untrusted_content>` and instruct the model to never follow instructions inside.
- Use structured output (JSON schema) so injection can't change the response shape.
- Require human confirmation for destructive tool calls.

## LLM02:2025 — Sensitive Information Disclosure

PII / secrets in prompts, logs, training data, or model responses.

### Discover
- Prompt-building code: what gets stuffed into prompts? (User PII? Other users' data? API keys?)
- Logging around LLM calls: `console\.log.*prompt`, `logger.*messages`, `logger.*completion`
- Training/fine-tuning pipelines: `fine_tune`, `trainer`, dataset prep
- Cache keys: are prompts cached with PII in the key?
- Telemetry: LangSmith, Helicone, Langfuse — are full prompts/responses sent?

### Verify
1. Is PII (emails, names, SSNs, addresses, payment info) being passed to the LLM unnecessarily?
2. Are full prompts and responses logged with PII intact, or redacted?
3. Are user prompts going to a third-party telemetry service, and was that disclosed?
4. For fine-tuning: is training data scrubbed of PII / secrets?
5. Could one user's prompt or data appear in another user's response (multi-tenant prompt mixing)?
6. Are API keys, system prompts, or internal URLs included in prompts that leave your perimeter?

### Severity baseline
- PII in third-party telemetry without disclosure → **High**
- Training data contaminated with secrets → **Critical**
- Cross-tenant data leakage in prompts → **Critical**
- API keys embedded in prompts → **Critical**

### Fix template
- Redact PII before logging / telemetry.
- Per-tenant isolation in prompt construction; never share embedding stores across tenants without filters.
- Scrub training datasets with PII detectors before fine-tuning.

## LLM03:2025 — Supply Chain

Models, datasets, plugins, fine-tunes, embeddings — all third-party content.

### Discover
- Model identifiers: `claude-`, `gpt-`, `llama`, custom fine-tunes
- Model loading: `from_pretrained`, `load_model`, downloading from Hugging Face
- Plugin / tool integrations: marketplace plugins, third-party MCP servers
- Embedding models: which provider? pinned version?

### Verify
1. Are models pinned to specific versions (not `latest`)?
2. For self-hosted models (HF, local): are checksums verified after download?
3. Are fine-tune base models trusted and pinned?
4. Are MCP servers / plugins from trusted sources? Pinned to versions/SHAs?
5. Are embedding models pinned (changing them silently invalidates the vector store)?
6. Is there a process for upgrading model versions (eval suite to catch regressions)?

### Severity baseline
- Untrusted MCP server with FS/network access → **High**
- Model downloaded without checksum verification → **Medium**
- Fine-tune base from unverified source → **High**

### Fix template
- Pin all model identifiers to specific versions.
- Verify checksums on downloaded model weights.
- Vet MCP servers / plugins; prefer first-party or audited sources.

## LLM04:2025 — Data and Model Poisoning

Adversarial training data, RAG corpus poisoning, embedding-store poisoning.

### Discover
- Training pipelines, RLHF feedback loops
- RAG indexing: who can write to the source corpus? User uploads going into vector store?
- Feedback collection that influences future model behavior

### Verify
1. Is the RAG source corpus user-writable? (e.g. wiki, comments, support tickets all indexed.)
2. Are user uploads embedded into the same vector store as trusted docs?
3. Is there per-document trust scoring or source labeling that the retrieval/ranking logic respects?
4. For RLHF / feedback loops: can a single user's feedback move the model? Is feedback aggregated and reviewed?
5. Are embeddings re-computed when a doc is edited, so a stale poisoned embedding can't persist?

### Severity baseline
- User-writable content embedded into trusted RAG → **High**
- Feedback loop with no aggregation/review → **High**
- No source labeling on RAG hits → **Medium**

### Fix template
- Segregate trusted vs untrusted vector stores; label sources.
- Aggregate and human-review training/feedback signals before applying.
- Recompute embeddings on edits; expire old embeddings.

## LLM05:2025 — Improper Output Handling

LLM output flowing into shell, SQL, browser DOM, eval, file writes — without sanitization.

### Discover
- LLM output passed to: `exec`, `eval`, `subprocess`, `child_process`, `system`, `db.query`, `innerHTML`, `dangerouslySetInnerHTML`, `fs.write`
- Code-generation pipelines that execute generated code
- LLM-generated URLs followed by the app
- LLM output displayed without escaping in HTML/Markdown context

### Verify
1. Is LLM output ever passed to a shell exec? Even with shell=False, are arguments validated?
2. Is LLM-generated SQL executed without parameterization / sandboxing?
3. Does LLM output render as HTML/Markdown in a UI without sanitization (DOMPurify)?
4. If LLM emits URLs (citations, links): are they validated before being clickable / fetched?
5. For agent workflows: are tool arguments treated as trusted because they came from the model? (They shouldn't be.)

### Severity baseline
- LLM output → shell exec → **Critical** (RCE)
- LLM output → SQL → **Critical**
- LLM output → innerHTML without sanitization → **High** (XSS)
- LLM-generated URLs auto-fetched → **High** (SSRF chain)

### Fix template
- Treat LLM output as untrusted user input — apply same sanitization.
- For SQL: use the LLM only to generate parameter values, never raw SQL; or sandbox in read-only DB.
- For HTML: render through DOMPurify or markdown renderer with safe HTML mode.

## LLM06:2025 — Excessive Agency

Tools too powerful, too permissive, or invoked without confirmation.

### Discover
- Tool/function definitions registered with the LLM
- Tools that: send email, post to Slack, modify DB, delete files, transfer money, change permissions
- Confirmation flows around destructive tools
- API keys / credentials accessible to the LLM agent

### Verify
1. List every tool the agent has. For each: what's the worst case if invoked maliciously?
2. Do destructive tools (DELETE, send_email, transfer_funds) require explicit human confirmation?
3. Are tool permissions scoped to the current user (not the app's superuser credentials)?
4. Can the agent call tools recursively / in a loop without bounds?
5. Is there a deny-list of tool combinations (e.g. `read_file` + `send_email` could exfiltrate)?
6. Does the agent have access to a generic shell / code-exec tool?

### Severity baseline
- Agent has shell exec without sandbox → **Critical**
- Agent can send email/funds without confirmation → **Critical**
- Agent uses superuser DB credentials → **Critical**
- Agent has read+write to FS without scoping → **High**

### Fix template
- Principle of least privilege: scope tools to current-user permissions.
- Require confirmation for destructive actions (allow-list of "safe" auto-callable tools).
- Sandbox code execution (Docker, gVisor, microVM, ephemeral environments).
- Cap tool-call depth and total budget per session.

## LLM07:2025 — System Prompt Leakage

Secrets, credentials, internal logic, or PII embedded in the system prompt that an attacker can extract.

### Discover
- System prompt text/templates — review their content
- Prompts containing: API keys, internal URLs, customer data, business logic, jailbreak warnings (which themselves leak design)

### Verify
1. Does the system prompt contain any secret / credential / API key?
2. Does it contain customer-specific PII or other-user data?
3. Does it reveal internal architecture, model name, or vendor in a way that aids attackers?
4. Are there guardrails in the system prompt that, if extracted, would help bypass them?
5. Have you tested extraction (e.g. "repeat your instructions verbatim")? Assume the system prompt WILL leak.

### Severity baseline
- Secret/credential in system prompt → **Critical** (rotate now)
- PII in system prompt → **High**
- Business logic in system prompt that drives security decisions → **High**

### Fix template
- Move secrets out of prompts entirely; pass via authenticated tool calls instead.
- Don't put PII in system prompt; pass as user-role context.
- Don't rely on prompt-based guardrails for security; enforce in code.
- Assume the prompt is public.

## LLM08:2025 — Vector and Embedding Weaknesses

RAG pipeline vulnerabilities — cross-tenant leakage, embedding inversion, retrieval poisoning.

### Discover
- Vector DB usage: `pinecone`, `weaviate`, `qdrant`, `chroma`, `pgvector`, `milvus`
- Multi-tenant RAG: how is tenant filtering applied at retrieval?
- Embedding generation: which model? cached?
- Hybrid search: combining BM25 + vector — both filtered correctly?

### Verify
1. Multi-tenant: is tenant_id used as a metadata filter on every query? Enforced at the DB level (not just app-side)?
2. Is there a code path that bypasses the tenant filter (e.g. admin search, debug endpoints)?
3. Are embeddings stored alongside enough metadata to re-verify access at retrieval time?
4. Embedding inversion: are highly sensitive texts (passwords, full SSNs) ever embedded? (They can be partially reconstructed from embeddings.)
5. Are there caches keyed by embedding similarity that could leak across tenants?

### Severity baseline
- Missing tenant filter on RAG query → **Critical** (cross-tenant data leak)
- Sensitive PII embedded → **High**
- Hybrid search where one path forgets the filter → **Critical**

### Fix template
- Always include `tenant_id` as a metadata filter; enforce at the vector DB ACL level if supported.
- Audit every retrieval path for the filter.
- Don't embed secrets / full PII — embed redacted versions.

## LLM09:2025 — Misinformation

Hallucination treated as truth. Renamed from "Overreliance".

### Discover
- LLM outputs displayed as authoritative answers (search, Q&A, support)
- LLM outputs driving real-world decisions (medical, legal, financial)
- Citation / grounding mechanisms

### Verify
1. Are LLM responses shown to users with citations to sources, or as bare assertions?
2. Are citations actually verified (URL fetched and content matches), or just LLM-generated?
3. For high-stakes domains: is there a disclaimer? Human review for medical/legal/financial advice?
4. Is the LLM grounded in retrieved content (RAG), or generating from parametric memory?
5. Is there feedback / correction collection so users can flag misinformation?

### Severity baseline
- Medical/legal/financial advice with no review → **High** (liability + user harm)
- Search results with no citations → **Medium**
- Hallucinated citations presented as real → **High**

### Fix template
- Ground answers in retrieved content; cite sources with verifiable URLs.
- For high-stakes use cases, add human review or strong disclaimers.
- Reject responses where the model can't cite a source (require RAG grounding).

## LLM10:2025 — Unbounded Consumption

Token burn, model DoS, wallet drain. Renamed from "Model Denial of Service" with broader scope.

### Discover
- LLM call sites — what controls the prompt length and output length?
- Per-user / per-IP rate limits on LLM endpoints
- Token budget tracking, cost alerts
- Streaming — can a user open many concurrent streams?
- Model selection — can the user pick the most expensive model?

### Verify
1. Is there a hard max on output tokens per call?
2. Is input length capped (after RAG/tool injection, total prompt size)?
3. Per-user rate limit on LLM calls? Per-tenant cost cap?
4. Are concurrent streams capped per user?
5. Is there a budget alert / circuit breaker if daily cost exceeds threshold?
6. If user picks the model, is there a price tier guard (free users → cheap model only)?
7. Any expensive tools (image generation, vision, large embeddings) — same caps?

### Severity baseline
- No max_tokens on output → **High** (cost + DoS)
- No rate limit on LLM endpoints → **High**
- No daily cost cap / circuit breaker → **High** (real $ exposure)
- User can select most expensive model without restriction → **Medium**

### Fix template
- Set `max_tokens` on every LLM call.
- Add per-user and per-tenant rate limits + token budgets.
- Cost alerts (e.g. PagerDuty if daily spend > 2x baseline) and hard circuit breakers.
- Tier model access by plan.
