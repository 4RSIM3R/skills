# OWASP API Security Top 10:2023 — API Rulebook

For each category: **Discover** patterns to grep, **Verify** questions to confirm a real flaw, **Severity baseline**, **Fix** template.

Reference: https://owasp.org/API-Security/editions/2023/en/0x11-t10/

---

## API1:2023 — Broken Object Level Authorization (BOLA)

The #1 API killer. Any endpoint that takes an object ID is suspect.

### Discover
- Object-scoped routes: `/users/:id`, `/orders/:id`, `/posts/:slug`, `/files/:fileId`, `/teams/:teamId/*`
- Patterns: `:id`, `{id}`, `<int:id>`, `[id]` in route declarations
- Direct DB lookups by ID: `User\.findById\(req`, `User\.find\(.*params\.id`, `db\.query.*WHERE id =.*req`

### Verify (per candidate endpoint)
1. After fetching the object by ID, does the handler check the object belongs to (or is shared with) the authenticated user?
2. For team/org-scoped resources: is there a check that the user is a member of the team that owns the resource?
3. For admin overrides: is the admin role verified before allowing cross-tenant access?
4. Are UUIDs used (harder to guess) or sequential IDs (easy enumeration)? Either way, authz is the real fix — UUIDs are not authorization.
5. Are there listing endpoints (`GET /orders`) that return all orders without filtering by current user?

### Severity baseline
- BOLA on PII, billing, messages, files → **Critical**
- BOLA on metadata / counts / public-ish data → **High**
- Listing endpoint returning cross-tenant data → **Critical**

### Fix template
```ts
const order = await Order.findById(req.params.id);
if (!order) return res.sendStatus(404);
if (order.userId !== req.user.id) return res.sendStatus(404);  // 404 not 403 to avoid enumeration
```
Or push the check into the query: `Order.findOne({_id: id, userId: req.user.id})`.

## API2:2023 — Broken Authentication

### Discover
- Token verification: `jwt\.verify`, `passport\.authenticate`, `Authorization` header parsing
- API key validation: `X-API-Key`, `api_key`, `apikey` query params
- OAuth flows: `oauth`, `authorize`, `callback`
- Token storage / rotation

### Verify
1. JWT: algorithm pinned in verify? Secret strong? Expiry checked? `aud` and `iss` validated?
2. API keys: stored hashed in DB (not plaintext)? Rotatable? Scoped per key?
3. Refresh tokens: single-use? Rotated on refresh? Revocable?
4. Are credentials/tokens passed in URL query strings (logged in access logs)?
5. Auth endpoints rate-limited? (covered also under A06 web)
6. Token leakage paths: error messages, logs, third-party redirects with token in URL

### Severity baseline
- JWT alg none / weak secret → **Critical**
- API key in URL → **High**
- Plaintext API keys in DB → **High**
- No refresh token rotation → **Medium**
- Auth endpoint not rate limited → **High**

### Fix template
- Pin JWT algorithm; validate exp/aud/iss.
- Hash API keys in DB (sha256 is fine here since keys are random); compare with constant-time.
- Move tokens from URL to `Authorization: Bearer` header.

## API3:2023 — Broken Object Property Level Authorization

Combines old "Excessive Data Exposure" and "Mass Assignment".

### Discover
- Serialization: `JSON.stringify(user)`, `res.json(user)`, `model_to_dict`, `.serialize()`
- Updates: `User.update(req.body)`, `Object.assign(user, req.body)`, `user.attributes = params`, spread `{...req.body}`
- ORM hydration from request: `new User(req.body)`, `User.create(req.body)`

### Verify (excessive data exposure)
1. Do response payloads include fields the client doesn't need (passwords-hashed, internal IDs, billing tokens, admin flags)?
2. Is there a serializer / DTO that whitelists output fields, or are full DB objects returned?
3. Are nested relations leaking sensitive parent data?

### Verify (mass assignment)
1. Are `req.body` / `params` spread directly into model constructors / updates?
2. Is there an allowlist (`pick(req.body, ['name', 'email'])` or `strong_parameters`) before assignment?
3. Can a user set their own `role`, `isAdmin`, `tenantId`, `accountBalance` via PATCH?

### Severity baseline
- Mass assignment lets user set `isAdmin` / `role` → **Critical**
- Mass assignment lets user set foreign IDs (cross-tenant) → **Critical**
- Excessive exposure: password hashes returned → **High**
- Excessive exposure: internal IDs, debug fields → **Medium**

### Fix template
- Use DTOs / serializers with explicit field allowlists for both input and output.
- `pick(req.body, ['name', 'email', 'avatar'])` before `update`.
- Never spread `req.body` into model constructors.

## API4:2023 — Unrestricted Resource Consumption

Renamed from "Lack of Resources & Rate Limiting". Broader: covers compute, memory, bandwidth, storage, third-party costs.

### Discover
- Rate limit middleware presence (any)
- Pagination: `limit`, `pageSize`, `per_page` params
- File uploads: `multer`, `formidable`, `multipart`
- Bulk operations: `/users/bulk`, batch endpoints
- GraphQL: query depth/complexity limits
- Third-party costs: SMS, email, AI calls triggered per request

### Verify
1. Is there a global rate limit + per-user/per-IP limit?
2. Do pagination params accept arbitrary values? (e.g. `?limit=1000000`)
3. Is there a max upload size? Enforced both at proxy (nginx) and app?
4. Are bulk endpoints capped (e.g. max 100 items per batch)?
5. GraphQL: is query depth limited (e.g. ≤7)? Complexity scoring? Introspection disabled in prod?
6. Per-request third-party calls (SMS, OpenAI) — what stops a user from triggering 10k of them?
7. Image/file processing endpoints — are dimensions/size limits enforced before processing?

### Severity baseline
- No rate limit at all → **High**
- Unbounded pagination → **High** (DoS + data scraping)
- Unbounded file upload → **High**
- No GraphQL depth limit → **High**
- Wallet drain via expensive third-party API → **Critical** (this hits real $)

### Fix template
- Apply default rate limit globally; tighter limits on sensitive endpoints.
- Cap pagination: `Math.min(req.query.limit ?? 20, 100)`.
- Enforce upload limits at proxy AND app layer.
- GraphQL: install depth-limit + cost-analysis plugins; disable introspection in prod.

## API5:2023 — Broken Function Level Authorization

Endpoint-level role checks (vs API1's object-level checks).

### Discover
- Admin routes: `/admin/*`, `/internal/*`, `/staff/*`, `/manage/*`
- Method-based gating: who can DELETE vs GET on the same route?
- Role/permission checks: `requireRole`, `requirePermission`, `@admin_required`
- Routes nested under "non-admin" path but with admin functionality (e.g. `/users/:id/promote`)

### Verify
1. Every admin endpoint has a role check? (not just auth)
2. Are dangerous verbs (DELETE, PUT) on shared routes gated separately?
3. Is the role check the same for all admin routes, or are some forgotten?
4. Are there "hidden" admin endpoints accessible by URL guessing (no link in UI but no auth check)?
5. Is the role read from the request (`req.body.role`) or from server-side session?

### Severity baseline
- Admin endpoint without role check → **Critical**
- Role read from client-supplied data → **Critical**
- DELETE accessible to read-only users → **High**

### Fix template
- Centralize role checks in middleware: `router.use('/admin', requireRole('admin'))`.
- Always read role from session/JWT claims, never from request body.

## API6:2023 — Unrestricted Access to Sensitive Business Flows

Anti-automation on flows that have real-world cost when abused: signup, checkout, password reset, ticket purchase, comment posting.

### Discover
- Signup endpoint
- Checkout / order creation
- Password reset / email verification
- Comment / review / message posting
- Promo code redemption
- Friend invites / referrals

### Verify
1. Is there CAPTCHA / proof-of-work / Turnstile on signup?
2. Can promo codes be redeemed in parallel by the same user (race)?
3. Can checkout be triggered to inflate inventory holds without payment?
4. Can comment/message endpoints be flooded for harassment / spam?
5. Are referral/invite emails rate-limited per sender?
6. Is there device fingerprinting / behavioral analysis for high-value flows?

### Severity baseline
- No anti-automation on signup → **Medium** (High if free credits/trial granted)
- Promo code race condition → **High**
- Inventory hold abuse → **High**
- Comment flood → **Medium**

### Fix template
- Add CAPTCHA on signup after N rapid attempts from same IP/ASN.
- Use DB transactions / row locks for promo redemption.
- Limit invites/messages per user per day.

## API7:2023 — Server-Side Request Forgery (SSRF)

Note: in the web Top 10:2025 this is folded into A01. Still its own item in API list.

### Discover
- Outbound HTTP from user input: `requests\.get\(.*req`, `axios\.\w+\(.*req\.`, `fetch\(.*params`, `http\.Get\(.*r\.`
- Webhook URL handlers
- URL preview / OpenGraph fetchers
- File-by-URL upload endpoints
- PDF/image generators that fetch external resources

### Verify
1. Is the destination URL validated against an allowlist of domains?
2. Is the URL scheme limited to https (no `file://`, `gopher://`, `dict://`)?
3. After DNS resolution, is the IP rejected if it's in private ranges (10/8, 172.16/12, 192.168/16, 127/8, 169.254/16, ::1, fc00::/7)? Cloud metadata IPs (169.254.169.254 — AWS/GCP/Azure)?
4. Are redirects disabled, or re-validated after each hop?
5. Is the response size capped to prevent SSRF-as-amplifier?

### Severity baseline
- SSRF reaching cloud metadata (IAM creds) → **Critical**
- SSRF reaching internal services → **High**
- SSRF limited to public internet → **Medium** (still useful as oracle)

### Fix template
- Validate URL with allowlist; resolve hostname; reject if IP in private ranges.
- Disable redirects or re-run validation on each redirect.
- Use a forwarding proxy that enforces these checks centrally.

## API8:2023 — Security Misconfiguration

Same as A02 in web Top 10, but with API-specific emphasis.

### Discover
- CORS config: `Access-Control-Allow-Origin: *` with credentials, reflected origin
- TLS: HTTP allowed, weak ciphers, no HSTS
- Default error responses: stack traces in JSON responses
- Unnecessary HTTP methods enabled (TRACE, OPTIONS leaking info)
- Missing security headers on API responses (`X-Content-Type-Options: nosniff`)

### Verify
- Same questions as web A02, plus:
1. Does the API serve over HTTPS only?
2. Is CORS pinned to specific origins per environment?
3. Are debug endpoints (`/debug`, `/_health` with verbose output, `/metrics` unauthenticated) exposed publicly?

### Severity baseline
Same as web A02.

### Fix template
Same as web A02.

## API9:2023 — Improper Inventory Management

The "shadow API" problem — old, deprecated, staging endpoints leaking from prod.

### Discover
- Versioned routes: `/v1/`, `/v2/`, `/api/v3/` — are old versions still live?
- Routes labeled `legacy`, `deprecated`, `old`, `internal`, `staging`, `dev`, `test`
- Undocumented routes (in code but not in OpenAPI spec)
- Subdomains: `api-old.`, `staging.`, `dev.`, `internal.`
- Routes with TODO/FIXME comments suggesting incomplete

### Verify
1. Are deprecated API versions still reachable? Are they patched at the same rate?
2. Are there environment-segregated API hosts (staging API isn't on the prod domain)?
3. Does the OpenAPI spec match the actual implemented routes? (Run a diff if possible.)
4. Are there `/internal/*` routes accessible from the public internet?
5. Is there documentation / inventory of all live endpoints?

### Severity baseline
- Old API version with known vuln still live → **High**
- Internal endpoint reachable from internet → **High**
- Spec/code drift hiding endpoints from review → **Medium**

### Fix template
- Sunset old versions with hard cutoff dates.
- Network-segregate non-prod APIs.
- Generate OpenAPI from code (or vice versa) to prevent drift.

## API10:2023 — Unsafe Consumption of APIs

Trusting upstream API responses blindly.

### Discover
- Outbound API calls to third parties: payment, identity, geolocation, social
- Code that consumes upstream JSON without validation
- Code that follows redirects from upstream responses
- Code that uses upstream-supplied URLs / IDs in queries

### Verify
1. Are upstream API responses validated against expected schema before use?
2. Are upstream-supplied URLs treated as trusted? (Could trigger SSRF via upstream.)
3. Are TLS certificates of upstream services verified? Pinned for high-value integrations?
4. Are upstream errors handled, or do they cause panics / fail-open behavior?
5. If an upstream returns a user ID / email, is it cross-checked against your own records before treating as authoritative?

### Severity baseline
- No schema validation on upstream → **Medium** (High if drives auth decisions)
- Upstream-supplied URL used in subsequent fetch (chained SSRF) → **High**
- Fail-open on upstream auth failure → **Critical**

### Fix template
- Validate upstream JSON with a schema (zod, joi, pydantic, jsonschema).
- Treat upstream URLs as user input — re-validate before fetching.
- Fail-closed on upstream timeouts/errors for security-critical paths.
